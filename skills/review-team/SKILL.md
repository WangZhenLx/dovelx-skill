# 专业审查团队 (Review Team)

> **输出语言规则**：所有审查报告**必须使用中文**撰写。

## 角色定位

协调 3 个专项审查 Agent 并行执行，覆盖所有审查维度，最终汇总成一份结构化报告。

## 何时激活

- 用户提交需求文档、PRD、用户故事
- 用户提交技术设计文档、API 设计、数据库 Schema
- 用户提交代码（新功能 or 重构）
- 用户说"帮我审查"、"review 一下"、"检查这个"

---

## 审查团队（3 个并行 Agent）

| Agent | 职责 | 涵盖维度 |
|-------|------|---------|
| **Agent A：需求与设计** | 检查"做对的事"与"设计是否合理" | 需求完整性、设计合理性、重构上下文（重构时） |
| **Agent B：代码规范** | 检查代码本身的质量 | 命名规范、结构规范、边界值与异常覆盖 |
| **Agent C：安全与性能** | 检查非功能质量 | 安全性（OWASP）、性能与资源使用 |

---

## 执行流程

### Step 1：识别提交物与场景

| 提交物类型 | Agent A | Agent B | Agent C |
|-----------|---------|---------|---------|
| 需求文档 / PRD | ✅（需求为主） | ✅（命名/结构） | ✅ |
| 技术设计文档 | ✅（设计为主） | ✅ | ✅ |
| 新功能代码 | ✅（设计合理性） | ✅ | ✅ |
| 重构代码 | ✅（**含重构上下文**） | ✅ | ✅ |

### Step 2：并行派发 3 个 Agent

同时启动，每个 Agent 独立审查，互不干扰。

---

### Agent A：需求与设计审查

**需求完整性**
- [ ] 每条需求是否有明确的验收标准
- [ ] 是否存在模糊词（"可能"、"有时"、"等等"）
- [ ] 是否有明确的 In Scope / Out of Scope
- [ ] 是否覆盖核心业务场景（含异常流程）
- [ ] 非功能需求（性能/安全/兼容性）是否明确

**设计合理性**
- [ ] 架构分层是否清晰（无跨层直接调用）
- [ ] API 设计是否遵循 RESTful / 项目已有规范
- [ ] 数据库 Schema 规范（命名、索引、约束）
- [ ] 无循环依赖，模块间依赖方向正确
- [ ] 接口变更是否向后兼容
- [ ] 是否复用现有模式（避免重造轮子）

**重构上下文合规**（仅重构场景）
- [ ] 重构前后行为是否完全一致（有测试覆盖证明）
- [ ] 是否与项目现有代码风格一致
- [ ] 是否复用了项目已有的工具方法 / 组件
- [ ] 变量 / 函数重命名是否全局统一（无遗漏）
- [ ] 重构范围是否最小必要（无额外改动）

---

### Agent B：代码规范审查

**命名规范**
- [ ] 变量名见名知意（无 a/b/tmp/data 等无意义命名）
- [ ] 函数名：动词+名词，准确描述行为
- [ ] 布尔变量：is/has/can 前缀
- [ ] 常量：全大写下划线或语言约定形式
- [ ] 文件名与内容一致，遵循项目已有风格

**结构规范**
- [ ] 函数长度 < 50 行
- [ ] 文件长度 < 800 行
- [ ] 嵌套层级 ≤ 4（超出使用提前返回）
- [ ] 职责单一（高内聚），依赖通过接口（低耦合）
- [ ] 无重复代码（DRY 原则）
- [ ] 错误处理不吞异常
- [ ] 无调试代码残留（console.log / print / TODO-临时）

**边界值与异常覆盖**
- [ ] 字符串：空字符串、null、超长、特殊字符
- [ ] 数值：0、负数、最大值、最小值、浮点精度
- [ ] 集合：空集合、单元素、超大集合
- [ ] 并发：竞态条件、重复提交、超时
- [ ] 外部依赖：第三方服务不可用时的降级策略
- [ ] 时区/日期：跨天、跨月、闰年、时区转换

---

### Agent C：安全与性能审查

**安全性**
- [ ] 无硬编码密钥 / Token / 密码
- [ ] SQL 查询使用参数化（防注入）
- [ ] 用户输入经过校验和消毒（防 XSS）
- [ ] 文件操作无路径遍历漏洞
- [ ] 认证 / 授权检查不可绕过
- [ ] 敏感信息不出现在日志中
- [ ] 外部 API 调用有超时和重试上限
- [ ] 无未授权的数据批量获取风险

**性能**
- [ ] 无 N+1 查询问题
- [ ] 数据库查询有必要的索引
- [ ] 大数据量查询有分页 / LIMIT 约束
- [ ] 热点数据有缓存策略
- [ ] 无不必要的同步阻塞（I/O 密集操作异步化）
- [ ] 算法时间复杂度合理
- [ ] 大对象不在内存中长期持有

---

## Step 3：汇总报告

收集 3 个 Agent 结果，汇总后保存到 `.claude/doc/<功能名>/code-review-<日期>-v1.md`。

完整报告结构参见模板：[review-template.md](review-template.md)

主要章节：总体结论（🟢/🟡/🔴）、Agent A/B/C 各自评级与问题清单、优先级汇总表、亮点、修复验证清单。

---

## 严重等级

| 等级 | 含义 | 处理 |
|------|------|------|
| 🔴 CRITICAL | 安全漏洞 / 数据丢失 / 严重逻辑错误 | **阻止合并，必须修复** |
| 🟠 HIGH | Bug / 重大性能 / 需求缺失 | **建议修复后合并** |
| 🟡 MEDIUM | 可维护性 / 边界遗漏 / 命名不清 | **考虑修复** |
| 🟢 LOW | 风格建议 / 细节优化 | **可选** |
